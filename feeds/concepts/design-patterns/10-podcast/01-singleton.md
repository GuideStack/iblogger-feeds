## ១. កិច្ចសន្ទនា (Podcast Transcript)

**Host A (សិទ្ធិ - អ្នក​ជំនាញ):** សួស្តី​អ្នក​ទាំងអស់​គ្នា! ស្វាគមន៍​មក​កាន់​កម្មវិធី Engineering Podcast របស់​យើង។ ថ្ងៃ​នេះ​យើងនឹងជជែកគ្នាអំ​ពី **Singleton Pattern**។ លោក បូរិទ្ធ តើ​លោកធ្លាប់​មាន​បញ្ហា​ដែល​កម្មវិធី​ដំណើរ​ការ​យឺត ដោយសារ​តែ​បង្កើត Object Database Connection ច្រើនពេកទេ?

**Host B (បូរិទ្ធ - អ្នក​រៀន):** បាទសួស្តីបង! ធ្លាប់​តើ! កាលខ្ញុំរៀន​សរសេរ​កូដ​ដំបូង ឱ្យ​តែ​ពេល User ចុចប៊ូតុង ខ្ញុំ `new DatabaseConnection()` មួយ។ មាន​ពេល​មួយ User ចុចលេង ១០០០ ដង ម៉ាស៊ីន Server ខ្ញុំគាំងចេញផ្សែងហុយ ៗ តែ​ម្តង!

**Host A (សិទ្ធិ):** ហាហា! ហ្នឹងហើយ​គឺជា​បញ្ហា! ដើម្បី​ដោះស្រាយរឿង​នេះ យើងប្រើ Singleton។ ស្រមៃមើលធនាគារមួយ។ ធនាគារ​មាន​បុគ្គលិកបញ្ជររាប់សិបនាក់ ប៉ុន្តែ​ពួកគេ​មាន "ទូដែកធំ" តែ​មួយគត់។ គ្រប់​គ្នា​មិន​មែន​មាន​ទូដែកផ្ទាល់ខ្លួនឯងនៅ​ក្រោយ​តុ​នោះ​ទេ បើ​មិន​អញ្ចឹងទេ គ្មាន​នរណាដឹងថាធនាគារ​មាន​លុយ​ពិតប្រាកដ​ប៉ុន្​មាន​ឡើយ។ ដូច្​នេះ​ទូដែក​នោះ​គឺជា Single Source of Truth។

**Host B (បូរិទ្ធ):** ស្តាប់​ទៅ​ឡូយណាស់! ចុះ... សុំសួរឆ្កួត ៗ មួយបង! ចុះបើខ្ញុំ​ជា​បុគ្គលិកខិលខូច ខ្ញុំលួច Copy សោរទូដែកហ្នឹង ទៅ​វាយ​ធ្វើ​ទូដែកក្លែងក្លាយមួយទៀតនៅផ្ទះ តើ​ប្រព័ន្ធ​បង​មាន​វិធី​ការ​ពារម៉េច​ទៅ?

**Host A (សិទ្ធិ):** សំណួរឆ្កួតមែន តែ​ឆ្លាត! នៅក្នុង​កូដ ការ "វាយទូដែក​ថ្មី" គឺ​ស្មើនឹង​ការ​ហៅពាក្យ `new`។ ដើម្បី​ការ​ពារ​អ្នក​ខិលខូចដូចលោកបូរិទ្ធ យើង​ត្រូវ​ចាក់សោ `constructor` របស់ Class នោះ​ឱ្យ​ទៅ​ជា `private`។ នៅ​ពេល `constructor` ត្រូវ​បិទ ទោះលោកឯងខំវាយ `new DatabasePool()` រាប់ពាន់ដងទៀត ក៏ Compiler វៃក្បាលលោកឯងវិញដែរ! 

**Host B (បូរិទ្ធ):** អូហូ! ចិត្ត​អាក្រក់​មែនទែន! ចុះបើបិទទ្វារជិតឈឹងអញ្ចឹង តើ​ខ្ញុំអាចយកលុយ​ពី​ទូដែកហ្នឹង​មក​ប្រើ​បាន​ម៉េច​ទៅ? ឬ​ក៏ឱ្យខ្ញុំអង្គុយមើល?

**Host A (សិទ្ធិ):** កុំ​ភ័យ! យើង​មាន​ច្រកទ្វារ V.I.P មួយ គឺ Method `public static getInstance()`។ ពេល​លោកឯងហៅវា វានឹងឆែកមើល៖ បើទូដែក​មិន​ទាន់​មាន វានឹងហៅ​ជា​ង​មក​សង់ទូដែក​ថ្មី​មួយ។ តែ​បើ​មាន​រួចហើយ វាគ្រាន់​តែ​ចង្អុល​បង្ហាញ​ផ្លូវឱ្យលោកឯងដើរ​ទៅ​ប្រើទូដែក​ចាស់​នោះ​ជា​ការ​ស្រេច។

**Host B (បូរិទ្ធ):** ហាហា... ចុះបើខ្ញុំហៅមិត្តភក្តិ ១០០ នាក់ (Threads) ឱ្យរត់ចូល​ទៅ​ច្រកទ្វារ V.I.P ហ្នឹងព្រមគ្នា​ក្នុង​វិនាទី​តែ​មួយ (Race Condition) ម៉េចដែរ? ជា​ងសង់ទូដែកច្បាស់​ជា​វង្វេង សង់ទូដែក ១០០ អោយពួកខ្ញុំ​មិន​ខាន!

**Host A (សិទ្ធិ):** នែ៎! នៅ​មាន​ល្បិចទៀត! ហ្នឹងហើយ​ជា​កន្លែង​ដែល Singleton អាចបែកធ្លាយ​ពិត​មែន។ ដើម្បី​ទប់ទល់នឹងក្បាច់ហៅគ្នា​មក​សម្រុក​នេះ នៅក្នុង Java យើងប្រើតិចនិក "Double-Checked Locking"។ វា​ដូចជា​មាន Security Guard ពី​រ​ជា​ន់អញ្ចឹង។ ជា​ន់ទី១ ឆែកមើលក្រៅទ្វារ ជា​ន់ទី២ ចាក់សោទ្វារ (synchronized) រួចឆែកម្តងទៀត។ ធានាថា​មក ១០០០ នាក់ព្រមគ្នាក៏ចូល​បាន​តែ​ម្តងម្នាក់ដែរ ហើយទូដែក​តែ​មួយគត់នឹង​ត្រូវ​បាន​សង់ឡើង​យ៉ាង​រឹងមាំ!

## ២. ការ​បកប្រែ​ទៅ​ជា​កូដ (Translating to Code)

```java
public class DatabasePool {
    // 1. volatile ការពារ Security Guard កុំឱ្យវង្វេង (ការពារ Reordering)
    private static volatile DatabasePool instance;

    // 2. private constructor ការពារបូរិទ្ធមិនឱ្យប្រើ new
    private DatabasePool() { }

    // 3. Double-Checked Locking សម្រាប់ទាញយក Object
    public static DatabasePool getInstance() {
        if (instance == null) { // ឆែកជាន់ទី១
            synchronized (DatabasePool.class) { // ចាក់សោទ្វារ
                if (instance == null) { // ឆែកជាន់ទី២ ការពារហ្វូង Thread
                    instance = new DatabasePool();
                }
            }
        }
        return instance;
    }
}
```

## ៣. គុណសម្បត្តិ និង​គុណវិបត្តិ (Pros & Cons)

**គុណសម្បត្តិ (Pros):**
- **ការ​ពារ​ការ​បំផ្លាញម៉ាស៊ីន:** ការ​ពារ​ការ​បង្កើត Object ដែល​ប្រើ Memory ច្រើនដ​ដែល ៗ ។
- **Single Source of Truth:** ធានាថា​កម្មវិធី​ទាំងមូល​ប្រើប្រាស់​ទីតាំង​តែ​មួយគត់។

**គុណវិបត្តិ (Cons):**
- **Global State អាក្រក់​ជូរ​ជា​តិ:** ពេល​ធ្វើ Unit Test ទូដែក​នេះ​វានៅ​ជា​ប់រហូត ធ្វើ​ឱ្យ Test មួយ​មាន​ឥទ្ធិពល​លើ Test មួយទៀត ដូចខ្មោចលងអញ្ចឹង។
- **ស​ត្រូវ​របស់ Microservices:** បើ Server អ្នក​មាន ៣ គ្រឿង អ្នក​នឹង​មាន Singleton ៣ មិន​មែនមួយទេ!

## ៤. ចំណុចសំខាន់ ៗ (Key Takeaways)

- **បញ្ហា:** កម្មវិធី​ត្រូវ​ការ​វត្ថុ​តែ​មួយគត់ ដើម្បី​ចៀសវាង​ការ​ខ្ជះខ្​ជា​យ។
- **ដំណោះស្រាយ:** បិទ `constructor` ឱ្យ​ទៅ​ជា `private` និង​ប្រើ `getInstance()` ជា​មួយ Double-Checked Locking។
